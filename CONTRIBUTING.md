# 贡献指南

非常欢迎你的贡献！为提高协作效率，请按照下面流程与规范提交更改。

## 提交流程（推荐）
1. Fork 本仓库到你的账号。
2. 在 fork 中创建分支：`git checkout -b feature/描述` 或 `fix/描述`。
3. 在分支中实现你的更改并为变更添加说明/示例（如有必要）。
4. 提交并推送分支：
   ```bash
   git add .
   git commit -m "feat: 简短描述你的改动"
   git push origin feature/描述
   ```
5. 在本仓库创建 Pull Request，PR 模板中请包含：
   - 变更目的与场景
   - 变更内容摘要
   - 测试/验证步骤
   - 是否影响向后兼容或需要特殊合并策略

## 分支与提交规范
- 分支命名示例：
  - feature/xxx、fix/xxx、chore/xxx、docs/xxx
- 推荐使用简洁的 commit message：
  - feat: 增加新功能
  - fix: 修复 bug
  - docs: 文档变更
  - chore: 其它杂项（构建、依赖等）
- 每次提交应关注单一变更点，便于审查与回滚。

## 代码/配置质量要求
- 配置变更请在本地/测试环境验证，确保语法与目标平台兼容。
- 提交前请移除多余的调试注释或私有信息（如 API key、密码、敏感 IP 等）。

## PR 审查流程
- 提交 PR 后，维护者会进行审查，可能会要求补充说明或修改。
- 请对审查意见及时响应与更新代码，保持沟通礼貌与专业。

## 问题报告（Issue）
- 提 issue 时请提供：
  - 清晰的复现步骤
  - 目标环境（mihomo 版本 / OpenClash 版本 / 相关平台）
  - 日志片段或错误信息（如适用）
- 若你自己已提供修复方案，欢迎创建 PR 并在 issue 中链接。

## 许可与版权
- 提交内容即表示你授权本仓库以仓库 LICENSE（若存���）下使用你的贡献。
- 若提交来自第三方来源，请在 PR 中注明来源与许可信息，确保合规。

## 联系方式
如有疑问，可通过仓库 Issue 或直接在 PR 中留言。

谢谢你的贡献与支持！
